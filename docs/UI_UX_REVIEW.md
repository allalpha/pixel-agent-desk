**대상 문서:** PRD_FINAL.md v2.0.0
**검토 범위:** 사용자 경험, UI 가이드라인, 인터랙션 디자인, 접근성, 반응형 설계

## 1. 검토 요약

### 전반적인 평가: 7.2/10

**평가 근거:**
- 디자인 시스템 기반은 잘 구축되어 있으나, 일관성과 완결성 측면에서 개선이 필요
- 기술적 우수성에 비해 사용자 경험 설계의 체계성이 부족
- 접근성 고려사항이 상세히 식별되어 있으나, 구현 계획이 구체적이지 않음

### 주요 강점 3가지

1. **명확한 시각적 계층 구조**
   - 색상 팔레트가 타입별(Main/Sub/Team)과 상태별로 체계적으로 정의됨
   - 폰트 크기 계층(7px-14px)이 명확하게 구분되어 가독성 확보
   - 컴포넌트 구조가 잘 정의되어 유지보수 용이

2. **상태 기반 인터랙션 설계**
   - 6개 상태(Waiting/Working/Thinking/Done/Help/Error)에 대한 명확한 시각적 정의
   - 상태 전이 규칙이 논리적으로 구조화됨
   - 애니메이션 가이드(호버, 클릭, 펄스)가 구체적으로 제시됨

3. **실시간 피드백 시스템**
   - 50ms 미만 반응속도로 즉각적인 피드백 제공
   - 타이머 시각화로 사용자 심리적 안정감 확보
   - 캐릭터 애니메이션으로 재미있는 사용 경험 제공

### 주요 약점 3가지

1. **접근성 구현 계획 부재**
   - ARIA 라벨, 키보드 포커스, 스크린 리더 지원이 식별만 되고 구현 계획 없음
   - 색상 의존도가 높아 색맹 사용자 배려 부족
   - 고대비 모드 지원 계획 미포함

2. **반응형 디자인의 불완전성**
   - 90px 고정 셀 크기로 화면 크기 비례 조절 미흡
   - 모바일 지원이 최소 크기(220x280px)로 인해 사실상 불가능
   - DPI 스케일링 미반영으로 고해상도 디스플레이에서 가독성 저하 우려

3. **사용자 워크플로우 검증 부족**
   - 시나리오가 기술적 구현 중심으로 작성되어 실제 사용자 행동 패턴 반영 부족
   - 에러 복구 시나리오가 구체적이지 않음
   - 멀티 에이전트 관리 시의 인지 부하 감소 전략 미흡

## 2. 사용자 경험 모순 식별

### 2.1 UX 설계상의 모순

#### 모순 1: 자동화 vs 사용자 통제
- **문제:** 자동 세션 복구는 Zero-config를 강조하지만, 복구 실패 시 사용자 피드백이 부족
- **영향:** 사용자가 무슨 일이 일어났는지 이해하지 못해 혼란 발생
- **위험도:** 높음 (High)
- **해결 제안:**
  ```
  - 복구 진행률 표시 (Progress Bar)
  - 실패 시 명확한 에러 메시지와 재시도 옵션 제공
  - 복구된 세션 수/실패한 세션 수 명시
  ```

#### 모순 2: 단순성 vs 정보 밀도
- **문제:** 7px 폰트(서브 배지)는 너무 작아 가독성 저하, 멀티 에이전트 모드에서 정보 과부하 우려
- **영향:** 사용자가 에이전트 상태를 신속하게 파악하기 어려움
- **위험도:** 중간 (Medium)
- **해결 제안:**
  ```
  - 최소 폰트 크기를 10px로 상향 조정
  - 정보 계층 구조 개선 (핵심 정보만 우선 표시)
  - 호버 시 확장 툴팁 제공
  ```

#### 모순 3: 즉각성 vs 성능
- **문제:** 50ms 미만 반응속도를 목표로 하지만, 동기 파일 I/O가 병목 지점
- **영향:** 실제 사용 경험에서 지연 발생, 애니메이션 부드러움 저하
- **위험도:** 높음 (High)
- **해결 제안:**
  ```
  - 비동기 I/O로 전환
  - 쓰기 버퍼링 도입
  - 로딩 상태 표시로 기다림의 고통 감소
  ```

### 2.2 사용자 워크플로우 문제

#### 문제 1: 터미널 포커스 실패 시 대안 부족
- **현황:** 캐릭터 클릭 → PowerShell 실행 → 터미널 포커스
- **문제점:** PID 조회 실패 시 대체 메커니즘 없음
- **사용자 경험:** 클릭했는데 아무 일도 일어나지 않는 것처럼 느껴짐
- **개선 제안:**
  ```javascript
  // 실패 시 피드백 제공
  if (!focusTerminal(pid)) {
    showUserMessage("터미널을 찾을 수 없습니다. 프로젝트 폴더를 엽니다.");
    openProjectFolder(projectPath);
  }
  ```

#### 문제 2: 멀티 에이전트 관리의 인지 부하
- **현황:** 최대 10개 에이전트 동시 표시
- **문제점:** 한눈에 모든 상태를 파악하기 어려움, 서브에이전트 관계 복잡
- **사용자 경험:** 정보 과부하로 중요한 에이전트를 놓칠 수 있음
- **개선 제안:**
  ```
  - 에이전트 필터링 기능 (프로젝트별, 상태별)
  - 즐겨찾기 에이전트 상단 고정
  - 서브에이전트 접기/펼치기 기능
  - 그룹별 요약 상태 표시
  ```

#### 문제 3: 에러 상태에서의 복구 가이드 부족
- **현황:** Error 상태는 빨간색으로 표시만 하고 구체적 조치 안내 없음
- **문제점:** 사용자가 무엇을 해야 할지 모름
- **사용자 경험:** 좌절감과 불안감 유발
- **개선 제안:**
  ```
  - 에러 유형별 구체적 메시지 제공
  - "재시도" 버튼 즉시 제공
  - "로그 보기" 링크로 상세 정보 확인 가능
  - 자동 재시도 옵션 (최대 3회)
  ```

### 2.3 인터페이스 불일치

#### 불일치 1: 색상 대비 기준 부재
- **문제:** WCAG AA/WCAG AAA 기준 명시 없이 주관적 색상 선정
- **예시:**
  - Working: #ff9800 (주황) - 배경과의 대비율 미측정
  - Thinking: #2196f3 (파랑) - 명도 대비 검증 필요
- **영향:** 저시력 사용자, 색맹 사용자 접근성 저하
- **해결 제안:**
  ```
  - 모든 상태 색상에 WCAG AA 준수 (최소 4.5:1)
  - 색상만으로 상태 구분하지 않고 아이콘/텍스트 병행
  - 색맹 시뮬레이션 테스트 수행
  ```

#### 불일치 2: 애니메이션 속도 기준 부재
- **문제:**
  - 호버: 0.2s cubic-bezier
  - 펄스: 0.6s 또는 1s (상태에 따라 다름)
  - 일관성 있는 애니메이션 시스템 부재
- **영향:** 사용자가 리듬을 예측하기 어려움
- **해결 제안:**
  ```
  - 애니메이션 속도 토큰 정의
    - Fast: 0.15s (클릭, 호버)
    - Normal: 0.3s (상태 전환)
    - Slow: 0.6s (펄스, 로딩)
  - 모션 이징 함수 통일 (Material Design 가이드 준수)
  ```

#### 불일치 3: 레이아웃 전환의 갑작스러움
- **문제:** 단일 모드(Flexbox) ↔ 멀티 모드(CSS Grid) 전환 시 애니메이션 부족
- **영향:** 시각적 충격, 사용자가 혼란스러움
- **해결 제안:**
  ```css
  .agent-grid {
    transition: all 0.3s cubic-bezier(0.4, 0, 0.2, 1);
  }

  /* FLIP 애니메이션 기술 도입 고려 */
  ```

## 3. 수정 제안

### 3.1 즉시 수정 필요 항목 (긴급)

#### 1. 폰트 크기 하한선 설정
- **현재:** 7px (서브 배지)
- **문제:** 가독성 심각 저하
- **제안:** 최소 10px로 상향, 너무 작은 텍스트는 아이콘으로 대체
- **우선순위:** P0 (긴급)
- **예상 시간:** 1시간

#### 2. 색상 대비율 검증 및 수정
- **현재:** WCAG 기준 미준수 가능성 높음
- **문제:** 접근성 규정 미준수
- **제안:** 모든 텍스트/배경 조합 WCAG AA (4.5:1) 이상 확보
- **우선순위:** P0 (긴급)
- **예상 시간:** 4시간

#### 3. 키보드 포커스 시각적 표시 추가
- **현재:** 포커스 표시 없음
- **문제:** 키보드 사용자가 현재 위치 파악 불가
- **제안:**
  ```css
  :focus-visible {
    outline: 2px solid #2196f3;
    outline-offset: 2px;
    border-radius: 4px;
  }
  ```
- **우선순위:** P0 (긴급)
- **예상 시간:** 2시간

#### 4. 에러 상태 복구 가이드 구현
- **현재:** 에러 메시지만 표시
- **문제:** 사용자가 다음 행동을 알 수 없음
- **제안:** 재시도 버튼, 로그 보기 링크, 구체적 에러 메시지
- **우선순위:** P0 (긴급)
- **예상 시간:** 6시간

### 3.2 개선 제안 (중단기)

#### 1. 접근성 전면 개선 계획 수립
- **범위:**

  - 스크린 리더 테스트 및 수정
  - 키보드 내비게이션 완전 지원
  - 고대비 모드 지원
- **예상 시간:** 2주
- **우선순위:** P1

#### 2. 반응형 디자인 체계화
- **범위:**
  - 뷰포트 크기별 브레이크포인트 정의
  - 유동적 그리드 시스템 도입
  - DPI 스케일링 지원
  - 모바일 최소 지원 (320px)
- **예상 시간:** 1주
- **우선순위:** P1

#### 3. 정보 아키텍처 개선
- **범위:**
  - 에이전트 필터링 기능
  - 그룹별 요약 상태
  - 검색 기능
  - 사용자 정의 레이아웃 저장
- **예상 시간:** 2주
- **우선순위:** P1

#### 4. 애니메이션 시스템 정비
- **범위:**
  - 애니메이션 속도/이징 토큰 정의
  - 상태 전환 애니메이션 개선
  - "애니메이션 감소" OS 설정 반영
- **예상 시간:** 1주
- **우선순위:** P1

### 3.3 추가 필요 내용 (장기)

#### 1. 사용자 테스트 계획 수립
- **목적:** 실제 사용자 행동 데이터 수집
- **방법:**
  - 유저빌리티 테스트 (5-10명)

- **우선순위:** P2

#### 2. 디자인 시스템 문서화
- **목적:** 디자인 일관성 유지, 개발 효율성 향상
- **내용:**
  - 컴포넌트 라이브러리 (Storybook)
  - 색상/타이포그래피/간격 토큰
  - 사용성 가이드라인
  - 예시/안티 패턴
- **예상 시간:** 3주
- **우선순위:** P2

#### 3. 다크/라이트 모드 지원
- **목적:** 사용자 선호도 반영, 눈의 피로 감소
- **범위:**
  - 색상 테마 시스템 구축
  - 자동 전환 (시스템 설정 연동)
  - 사용자 선택 저장
- **예상 시간:** 2주
- **우선순위:** P2

#### 4. 다국어 지원 (i18n)
- **목적:** 글로벌 사용자 확보
- **범위:**
  - 영문, 한글, 일문 우선 지원
  - UI 텍스트 외부화
  - 번역 가이드라인
- **예상 시간:** 3주
- **우선순위:** P2

## 4. 타 전문가와의 협업 필요사항

### 4.1 기획팀과 조율 필요사항

#### 1. 페르소나 검증 및 구체화
- **현재 문제:** 파워 유저 개발자 페르소나가 너무 광범위
- **협업 필요:**
  - 실제 Claude CLI 사용자 인터뷰
  - 사용성 테스트 참여자 모집
  - 페르소나별 시나리오 검증
- **기대 결과:** 타겟 사용자 명확화, 기능 우선순위 재조정

#### 2. 기능 우선순위 재검토
- **현재 문제:** 접근성 개선이 P2로 분류되어 있음
- **협업 필요:**
  - 비즈니스 가치 vs 사용자 경험 균형 논의
  - 법적 요구사항(접근성) 검토
  - 경쟁 제품 대비 차별화 포인트 확인
- **기대 결과:** 접근성을 P0로 상향 조정

#### 3. 성공 지표 설정 공유
- **현재 문제:** UI/UX 관련 지표 부족 (NPS, CSAT만 있음)
- **협업 필요:**
  - SUS (System Usability Scale) 측정 계획
  - Task Completion Rate 정의
  - Time on Task 기준선 설정
- **기대 결과:** 정량적 UX 지표 체계 확립

### 4.2 개발팀과 조율 필요사항

#### 1. 성능 최적화 협업
- **현재 문제:** 동기 I/O가 UI 응답성 저하
- **협업 필요:**
  - 비동기 I/O 전환 기술적 난이도 평가
  - 애니메이션 최적화 (requestAnimationFrame 도입)
  - 메모리 누수 수정 방법 논의
- **기대 결과:** 60fps 안정적 애니메이션, <100ms 상태 전환

#### 2. 접근성 구현 기술 지원
- **현재 문제:** ARIA 라벨, 키보드 내비게이션 구현 방법 미정
- **협업 필요:**
  - Electron 앱에서의 스크린 리더 테스트 방법 공유
  - 키보드 이벤트 핸들링 구현 가이드
  - 자동화 접근성 테스트 도구(axe-core) 도입
- **기대 결과:** WCAG 2.1 AA 준수

#### 3. 반응형 구현 기술 제약 확인
- **현재 문제:** 데스크톱 전용 Electron 앱에서 반응형 필요성 논의
- **협업 필요:**
  - 현재 구조에서 모바일 지원 가능성 검토
  - DPI 스케일링 구현 방법 공유
  - 윈도우 크기 조절 최적화 방안 논의
- **기대 결과:** 다양한 디스플레이 환경 지원

#### 4. 디자인 토큰 시스템 구축
- **현재 문제:** 하드코딩된 색상/크기 값이 코드에 산재
- **협업 필요:**
  - CSS 변수/JS 객체로 토큰화 방법 논의
  - 디자인-개발 간 토큰 동기화 방안 (Figma Tokens 등)
  - 테마 전환 아키텍처 설계
- **기대 결과:** 디자인 일관성 유지, 개발 효율성 향상

### 4.3 QA팀과 조율 필요사항

#### 1. 접근성 테스트 계획 수립
- **현재 문제:** 접근성 테스트가 QA 계획에 포함되지 않음
- **협업 필요:**
  - 자동화 테스트 도구(axe-core, WAVE) 도입
  - 수동 테스트 체크리스트 작성
  - 스크린 리더 테스트 시나리오 개발
- **기대 결과:** 각 빌드마다 접근성 자동화 테스트

#### 2. 크로스 브라우저/플랫폼 테스트
- **현재 문제:** Windows 전용으로 타 플랫폼 테스트 부족
- **협업 필요:**
  - macOS/Linux 테스트 환경 구축
  - 다양한 DPI 설정 테스트 시나리오
  - 고대비 모드 테스트 방법 공유
- **기대 결과:** 모든 지원 플랫폼에서 일관된 경험

#### 3. UI 회귀 테스트 자동화
- **현재 문제:** UI 변경 시 시각적 회귀 감지 어려움
- **협업 필요:**
  - Visual Regression Testing 도구 도입 (Percy, Chromatic)
  - 스크린샷 기반 테스트 시나리오 작성
  - 애니메이션 정확성 테스트 방법 논의
- **기대 결과:** UI 변경 시 자동화된 시각적 회귀 감지

#### 4. 사용성 테스트 지원
- **현재 문제:** 실제 사용자 관찰 테스트 계획 미수립
- **협업 필요:**
  - 유저빌리티 테스트 세션 운영 방법 논의
  - 사용자 행동 로깅 도구(없는 경우) 도입 검토
  - A/B 테스트 인프라 구축
- **기대 결과:** 데이터 기반의 UX 개선

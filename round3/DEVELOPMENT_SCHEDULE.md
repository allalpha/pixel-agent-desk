## 1. 기술 부채 우선순위

### 1.1 P0: 즉시 수정 (1주 내 완료)

**기준:** 사용자 경험에 직접적인 영향을 주거나, PRD 핵심 목표 달성을 방해하는 문제

#### **P0-1: 동기 I/O 제거 (8시간)**
- **문제:** `fs.appendFileSync`, `fs.writeFileSync`가 메인 스레드 차단
- **영향:** PRD 성능 목표 "이벤트 지연 < 100ms (P95)" 달성 불가
- **해결:** 비동기 로깅 시스템 구현 (배치 큐 + setTimeout)
- **담당:** Backend Developer (1일)
- **산출물:** `asyncLogger.js`

#### **P0-2: interval 정리 로직 (4시간)**
- **문제:** 정리되지 않은 interval로 메모리 누수
- **영향:** 장시간 실행 시 앱 충돌 가능성
- **해결:** intervals 배열로 관리하고 app.on('before-quit')에서 정리
- **담당:** Junior Developer (0.5일)
- **산출물:** `timerManager.js`

#### **P0-3: 접근성 긴급 개선 (8시간)**
- **문제:** 7px 폰트, 키보드 네비게이션 불가
- **영향:** PRD "접근성 기준" 위반, 장애인 사용자 배제
- **해결:** 최소 12px 폰트, ARIA 라벨 추가, 키보드 네비게이션
- **담당:** Frontend Developer (1일)
- **산출물:** 접근성 수정 이슈 목록

#### **P0-4: 에러 핸들링 개선 (10시간)**
- **문제:** 빈 catch 블록, 에러를 조용히 무시
- **영향:** PRD "충돌율 < 0.1%" 목표 달성 불가
- **해결:** 에러 바운더리 구현, 사용자에게 알림 표시
- **담당:** Senior Developer (1.3일)
- **산출물:** `errorHandler.js`

**P0 총 예상 시간:** 30시간 (약 4일)

### 1.2 P1: 1개월 내 완료

**기준:** 팀 협업 및 CI/CD 도입을 위해 필수적인 작업

#### **P1-1: main.js 모듈 분리 (16시간)**
- **문제:** 989줄 단일 파일, Git 병합 충돌 확률 90%+
- **영향:** 팀 병행 개발 불가
- **해결:** 5개 모듈로 분리 (WindowManager, HookServer, SessionManager, IpcHandlers, MissionControl)
- **담당:** Senior Developer (2일)
- **산출물:** `src/main/modules/` 디렉토리

#### **P1-2: 의존성 주입 리팩토링 (16시간)**
- **문제:** 전역 상태 과다, 테스트 불가능한 구조
- **영향:** 단위 테스트 작성 불가, CI/CD 도입 불가
- **해결:** 생성자에서 dependencies 객체 주입
- **담당:** Senior Developer (2일)
- **산출물:** DI 적용된 핵심 클래스들

#### **P1-3: Jest 설정 및 테스트 20% (24시간)**
- **문제:** 테스트 커버리지 0%
- **영향:** PRD "1개월 내 테스트 커버리지 20%" 목표
- **해결:** Jest 설정, 핵심 로직 단위 테스트 작성
- **담당:** QA + All Developers (3일)
- **산출물:** `__tests__/` 디렉토리, `.jest.config.js`

#### **P1-4: ESLint/Prettier 설정 (12시간)**
- **문제:** 코드 스타일 일관성 없음
- **영향:** 코드 리뷰 효율 저하
- **해결:** ESLint, Prettier 설정, 기존 코드 자동 수정
- **담당:** All Developers (1.5일)
- **산출물:** `.eslintrc.js`, `.prettierrc`, `package.json` 스크립트

**P1 총 예상 시간:** 68시간 (약 9일)

## 4. 합의안 제안

### 4.1 최소 합의 사항 (Must Have)

**다음 사항은 개발팀 전원이 동의해야 함:**

#### **합의 1: P0 수정은 1주 내 완료**
- 동기 I/O 제거 (8시간)
- interval 정리 로직 (4시간)
- 접근성 긴급 개선 (8시간)
- 에러 핸들링 개선 (10시간)

**이유:** 사용자 경험에 직접적인 영향, PRD 핵심 목표 달성 필수

#### **합의 2: P1 수정은 1개월 내 완료**
- main.js 모듈 분리 (16시간)
- 의존성 주입 리팩토링 (16시간)
- Jest 설정 및 테스트 20% (24시간)
- ESLint/Prettier 설정 (12시간)

**이유:** 팀 협업 가능성 확보, CI/CD 도입 필수

#### **합의 3: 신규 기능 개발은 P0+P1 완료 후 병행**
- P0 완료 전: 신규 기능 개발 중단 (긴급 버그 수정만)
- P1 완료 전: 신규 기능 개발 10% 이내

**이유:** 기술 부채가 심각하여 팀 협업 불가능 상태

---

### 4.2 타협안 (Trade-offs)

#### **타협안 1: P0와 P1을 병행하여 진행**

**주장:** P0만 먼저 완료하면 신규 기능 개발이 1주간 지연됨

**타협:**
- 1주차: P0-1, P0-2, P0-3만 완료 (24시간)
- 2주차: P0-4 완료 (10시간) + P1-1 시작 (16시간) + 신규 기능 개발 (14시간)

**위험:** P0-4(에러 핸들링)가 미완료 상태에서 신규 기능 추가 시 버그 증가 가능성

#### **타협안 2: 테스트 커버리지 목표를 낮춤**

**주장:** PRD의 "1개월 내 테스트 커버리지 20%"는 비현실적

**타협:**
- 1개월 차: 테스트 커버리지 10% (핵심 로직만)
- 2개월 차: 테스트 커버리지 20% (전체 기능)
- 3개월 차: 테스트 커버리지 30% (복잡한 로직)

**위험:** 버그 검증 능력 저하, regression bug 가능성

#### **타협안 3: P2를 6개월로 연기**

**주장:** P2(플랫폼 추상화)는 3개월이 아니라 6개월에 완료

**타협:**

## 5. 위험 요소

### 5.1 일정 위험 요소

#### **위험 1: 예상 시간 초과 (확률: 60%)**

**원인:**
- 기존 코드의 복잡성이 예상보다 높음
- 테스트 작성 시간이 예상보다 2배 소요
- 리팩토링 도중 버그 발견으로 수정 시간 증가

**대응:**
- 각 작업에 20% 버퍼 시간 추가
- P1 작업은 P0 완료 후 진행하여 일정 조정 가능
- 주차별 마일스톤 설정으로 진행 상황 모니터링

#### **위험 2: 인력 부족 (확률: 80%)**

**현실:**
- 1인 개발자가 1개월에 98시간(P0+P1) 작업은 불가능
- 2인 팀이라도 각자 49시간/주는 번아웃 위험

**대응:**
- **옵션 A:** 1인 개발 시 P1을 2개월로 연기
- **옵션 B:** 2인 팀 구성 (Senior Backend + Senior Frontend)
- **옵션 C:** 파트타임 Junior Developer 고용 (테스트 작성 지원)

#### **위험 3: 신규 기능 개발 지연 (확률: 100%)**

**현실:**
- P0+P1 완료까지 1개월 소요
- 이 기간 동안 신규 기능 개발은 10%에 불과

**대응:**
- PM과 협의하여 PRD 일정 조정
- 1개월 차는 "기술 부채 해결 전용"으로 공지
- P1 완료 후 신규 기능 개발 가속화

### 5.2 기술적 위험 요소

#### **위험 1: 리팩토링 중 기존 기능 파괴 (확률: 40%)**

**원인:**
- main.js 분리 중 IPC 통신 파손
- 의존성 주입 리팩토링 중 전역 상태 참조 오류
- 비동기 I/O 전환 중 레이스 컨디션 발생

**대응:**
- 각 리팩토링 전 regression 테스트 작성
- 기능 단위로 분리하여 PR 및 코드 리뷰
- 롤백 계획 수립 (git branch 전략)

#### **위험 2: 테스트 작성이 불가능한 구조 (확률: 70%)**

**원인:**
- Electron API에 강하게 결합된 코드
- 전역 상태에 의존하는 함수
- 의존성 주입이 안된 클래스

**대응:**
- 의존성 주입 리팩토링을 테스트 작성 전에 완료
- Electron API를 mock으로 분리
- 테스트 불가능한 코드는 리팩토링 대상으로 상정

#### **위험 3: 플랫폼 추상화 실패 (확률: 50%)**

**원인:**
- macOS Cocoa API에 대한 지식 부족
- Linux X11/Wayland 호환성 문제
- Windows 전용 로직이 분리되지 않음

**대응:**
- 플랫폼별 전문가 채용 또는 컨설팅
- Windows-only로 출시하고 나중에 포팅
- 오픈소스 커뮤니티에 기여 요청

### 5.3 팀 협업 위험 요소

#### **위험 1: Git 병합 충돌 (확률: 90%)**

**현실:**
- 2명 이상이 main.js를 동시에 수정 시 충돌 확정
- 충돌 해결 시간이 개발 시간보다 길어짐

**대응:**
- P1-1(main.js 분리)을 최우선으로 완료
- 모듈별 담당자 분배 (예: A는 WindowManager, B는 HookServer)
- Pull Request를 작게 자주 머지

#### **위험 2: 코드 리뷰 부재 (확률: 60%)**

**현실:**
- 1인 개발 시 코드 리뷰 불가능
- 2인 팀이라도 시간 부족으로 리뷰 건너뛰기

**대응:**
- Pull Request 크기를 200줄 이내로 제한
- 코드 리뷰를 개발 시간의 20%로 할당
- 리뷰어가 없을 경우 외부 리뷰어 요청

#### **위험 3: 지식 공유 부족 (확률: 80%)**

**현실:**
- 원작성자만 전체 구조를 이해
- 새 팀원이 onboarding되는 데 2주 이상 소요

**대응:**
- ARCHITECTURE.md 문서 작성 (P1 완료 후)
- 주요 함수에 JSDoc 주석 추가
- 주간 기술 세미나 진행
